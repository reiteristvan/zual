<objective>
Build the wall-clock progress engine at the heart of Zual's timer: an enum for the four
timer phases and a pure-Dart `TimerController extends ChangeNotifier` that derives a
normalized `progress` (0..1) and a `phase` from real elapsed wall-clock time, transitioning
to `done` exactly at completion regardless of animation/tick frame rate.

This plan delivers the setup → running → done portion of the state machine and the
drift-free progress calculation (TIMER-01, TIMER-02). Pause/resume, backgrounding
reconciliation, screen-wake, and app wiring are added in Plan 02.

Purpose: This is the foundation every later screen and scene consumes. Getting elapsed-time
math and the done transition correct — and proving it with deterministic tests — is the
highest-leverage work in the whole project.

Output: `lib/timer/timer_phase.dart`, `lib/timer/timer_controller.dart`,
`test/timer/timer_controller_test.dart`, and updated `pubspec.yaml`.

Timing-mechanism decision (Claude's Discretion per 01-CONTEXT.md): use `DateTime.now()`
wall-clock deltas as the elapsed-time source, NOT `Stopwatch`. Research recommended
`Stopwatch` for clean pause bookkeeping, but locked decision D-01 requires the countdown to
keep advancing while the app is backgrounded and the device may be asleep, and to be able to
reach `done` while backgrounded. A monotonic `Stopwatch` does not reliably count deep-sleep
time on Android; wall-clock `DateTime` deltas do, and match the design's own state model
(`startTs`/`pausedTotal`). See 01-PATTERNS.md, .planning/research/PITFALLS.md (Pitfalls 1-2),
and design/README.md state-management section.
</objective>

    - A new TimerController reports phase == setup and progress == 0.0.
    - start(5) with a fixed clock sets phase == running and progress == 0.0 at t0.
    - Advancing the injected clock to the halfway point yields progress ≈ 0.5 (within 0.01 tolerance).
    - Advancing the clock to or past total and calling syncToWallClock() yields phase == done and progress == 1.0.
    - progress reaching done is independent of how many times the ticker fired — asserting via manual clock advances plus a single syncToWallClock() call proves it is not tick-count-driven.

    - start() clamps its minutes argument into the inclusive range 1..120.
  </behavior>
  <action>
    Create lib/timer/timer_controller.dart with class TimerController extends ChangeNotifier.
    Imports are restricted to package:flutter/foundation.dart (for ChangeNotifier) and
    dart:async (for Timer) plus the local timer_phase.dart — no Material or Widgets imports, so
    the class stays unit-testable without a widget tree (this isolation is the single most
    important structural rule for the phase per ARCHITECTURE.md).

    Constructor: accept an optional injected clock of type `DateTime Function()` defaulting to
    DateTime.now, stored in a private field, and an optional tickInterval Duration defaulting
    to 200 milliseconds. The injected clock is what makes progress deterministic to test and is
    the elapsed-time source of truth (do not use Stopwatch).

    State fields (all private): the current TimerPhase (initial setup), the total Duration
    (initial Duration.zero), the run start timestamp as a nullable DateTime, a paused-time
    accumulator Duration (initial zero), a non-decreasing progress high-water mark double
    (initial 0.0), and a nullable Timer for the periodic ticker.

    Public getters: `TimerPhase get phase` returns the current phase; `double get progress`
    computes raw elapsed-fraction = elapsed-milliseconds / total-milliseconds, clamped to the
    0..1 range, and returns the maximum of that raw value and the stored high-water mark so
    progress is monotonic non-decreasing while a run is active. Guard the total==0 case to
    return 0.0 (avoid divide-by-zero). Define a private elapsed accessor that returns the clock
    delta since the start timestamp minus the paused accumulator, floored at Duration.zero so a
    backward clock movement cannot produce negative elapsed.

    start(int minutes): clamp minutes into 1..120; set total from the clamped minutes; record
    the start timestamp from the injected clock; reset the paused accumulator and the high-water
    mark to zero; set phase to running; start a Timer.periodic at tickInterval whose callback
    calls syncToWallClock; call notifyListeners.

    syncToWallClock(): make this a public method (the single reconcile path). Recompute progress;
    update the stored high-water mark to the larger of itself and the freshly computed raw
    fraction. If phase is running and elapsed is greater than or equal to total, transition phase
    to done, cancel and null the ticker, and set the high-water mark to 1.0. Always call
    notifyListeners at the end. This method exists so completion is a pure function of elapsed
    time — it will also be invoked on foreground return in Plan 02 (locked decision D-02:
    done is computed the moment the app returns to the foreground, even if it completed while
    backgrounded).

    Override dispose() to cancel the ticker before calling super.dispose() (leak discipline per
    PITFALLS Pitfall 4).

    Then create test/timer/timer_controller_test.dart as a plain unit test (import
    package:flutter_test/flutter_test.dart, package:zual/timer/timer_controller.dart,
    package:zual/timer/timer_phase.dart — no WidgetTester, no pumpWidget, no pumpAndSettle, no
    real delays). Use a mutable DateTime variable captured by a closure passed as the clock so
    the test advances simulated time instantly, then calls syncToWallClock() and asserts.
    Cover every case listed in the behavior block above, including the halfway-progress
    tolerance check, the done transition at completion, the tick-independence check, the
    monotonic-progress-under-backward-clock check, and the minutes-clamp bounds.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| device system clock → progress computation | Elapsed time is read from the device wall clock; a user (or NTP resync) can move the clock, which is untrusted input to the progress math. |
| pub.dev packages → app build | provider and wakelock_plus are third-party dependencies pulled at build time. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-01-01 | Tampering | progress getter / wall-clock elapsed (timer_controller.dart) | low | mitigate | Floor elapsed at Duration.zero and hold progress via a non-decreasing high-water mark so a backward device-clock change cannot rewind a running countdown. A forward clock jump only finishes the timer sooner — no data or privilege is at risk in a local kids' timer, so that case is accepted. |
| T-01-SC | Tampering | flutter pub add provider / wakelock_plus | low | mitigate | Pin exact caret versions from research (provider ^6.1.5 by the Flutter-team-adjacent maintainer, wakelock_plus ^1.6.1 by fluttercommunity), both long-established pub.dev packages sourced in STACK.md. pub is outside the npm/pip/cargo legitimacy-gate scope, so no blocking human checkpoint is required; verification is version-pinning plus successful `flutter pub get`. |

No Spoofing, Repudiation, Information Disclosure, or Elevation threats apply: this phase has no
network, no auth, no accounts, and no persisted data (ASVS L1, local-only engine).
</threat_model>
