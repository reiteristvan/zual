<objective>
Complete the timer state machine: add pause/resume (excluding paused time), backgrounding
reconciliation (the countdown keeps advancing against real wall-clock time and can reach done
while backgrounded), keep the screen awake while running, and wire a single TimerController
into the app root so the engine runs end-to-end.

This plan delivers TIMER-03 (pause/resume), TIMER-04 (backgrounding/foregrounding), TIMER-05
(screen stays awake), and completes TIMER-01 (the paused sub-state). It builds directly on the
wall-clock progress engine from Plan 01.

Purpose: These are the behaviours that make Zual feel like a real physical timer — pausing for
a snack and resuming, or locking the phone and coming back to a countdown that reflects real
elapsed time. They are proven with deterministic clock-injected tests (locked decision:
engine-only, no debug UI — correctness is demonstrated by the test suite, not a visible screen).

Output: pause/resume/endTimer + backgrounding reconcile on TimerController, a ScreenWake
abstraction with a wakelock_plus adapter, a lifecycle binder, updated main.dart, and expanded
tests.

Locked decisions honored: D-01 (no auto-pause on backgrounding — the Stopwatch-free wall-clock
model already keeps advancing), D-02 (done is computed on foreground return via syncToWallClock),
D-03 (no in-progress timer state persisted — nothing written to shared_preferences here).
</objective>

  <behavior>
    - pause() from running sets phase == paused and freezes progress (progress unchanged as the clock keeps advancing while paused).
    - pause() is a no-op when phase is not running; resume() is a no-op when phase is not paused.
    - resume() from paused sets phase == running and excludes the paused interval: start(5), advance 1 min, pause, advance 2 min (paused), resume, advance 4 min → phase == done (total active time 1+4 == 5), and the timer finished 2 minutes later in wall-clock terms than an uninterrupted run.
    - Backgrounding: start(5), advance the clock 5 min with NO ticks, then a single syncToWallClock() → phase == done and progress == 1.0 (done reached while "backgrounded", D-02).
    - Backgrounding mid-run: start(10), advance 4 min with no ticks, syncToWallClock() → phase still running and progress ≈ 0.4 (no reset, no drift, D-01).
    - endTimer() from any phase sets phase == setup and progress == 0.0.
  </behavior>
  <action>
    Extend TimerController (do not rewrite Plan 01's progress math). Introduce the paused-time
    model on top of the existing start-timestamp + pausedTotal fields already present.

    Add a private nullable "paused-at" DateTime field. Update the private elapsed accessor so
    that when phase is paused it measures up to the paused-at instant (freezing elapsed) rather
    than the live clock; when running it measures to the live injected clock. The result is that
    elapsed excludes paused wall-clock time — this is why no manual Stopwatch bookkeeping is
    needed and why backgrounding "just works" (elapsed is always derived from real timestamps).

    pause(): no-op unless phase == running. Record the paused-at instant from the injected clock,
    set phase to paused, then call notifyListeners. Per D-01 there is no lifecycle auto-pause —
    pause() is only ever an explicit parent action; backgrounding must NOT call it.

    resume(): no-op unless phase == paused. Add (injected-clock now minus paused-at) to the
    pausedTotal accumulator, clear the paused-at field, set phase to running, ensure the periodic
    ticker is running again (restart it if it was cancelled), then call notifyListeners.

    endTimer(): cancel and null the ticker; reset total to Duration.zero, clear the start
    timestamp and paused-at, reset pausedTotal and the progress high-water mark to zero; set phase
    to setup; call notifyListeners. Do NOT persist anything (D-03).

    Confirm syncToWallClock() (from Plan 01) already flips running → done when elapsed >= total;
    this is what makes done-while-backgrounded work when it is next called on foreground return.
    No AppLifecycleState code lives in the controller itself — the controller stays pure
    (foundation + dart:async only); the resume hook is wired in Task 2.

    Extend test/timer/timer_controller_test.dart with the behavior-block cases above. Continue to
    use the injected mutable clock and direct syncToWallClock() calls — never real delays, never
    pumpAndSettle. The backgrounding cases specifically advance the clock WITHOUT letting any tick
    fire, then call syncToWallClock() once, to model a throttled/suspended background.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| app lifecycle events → timer reconciliation | The OS delivers background/foreground transitions; the app reconciles timer state on resume. |
| running-phase lifetime → device screen-wake resource | Wakelock is a system resource that must be released; a stuck wakelock drains battery on an unattended device. |
| process kill → in-memory timer state | A running timer exists only in memory; an OS process kill discards it (D-03). |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-01-02 | Tampering | pause/resume/start/endTimer transitions (timer_controller.dart) | low | mitigate | Each transition is a no-op unless the current phase is valid (pause only from running, resume only from paused), preventing pausedTotal/elapsed corruption from out-of-order calls; asserted by unit tests. |
| T-01-03 | Denial of Service | Timer.periodic ticker + ScreenWake lifecycle | medium | mitigate | dispose() cancels the ticker; ScreenWake.disable() is called on every non-running transition (pause, done, endTimer) so the screen can never be held awake after a run; unit tests assert disable-on-done and disable-on-end. |
| T-01-04 | Information Disclosure / Tampering | app-kill recovery (D-03) | low | accept | No in-progress timer state (start time, duration, phase) is persisted; a process kill returns the app to setup. This is the intended behavior and leaves no at-rest state to protect or tamper with. |

No Spoofing, Repudiation, or Elevation threats apply (ASVS L1, no network, no auth, no accounts,
no persisted sensitive data).
</threat_model>
